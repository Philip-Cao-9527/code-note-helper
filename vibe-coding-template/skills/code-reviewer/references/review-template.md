# Code review prompt 模板

本文件是 `code-reviewer` 的专属参考模板。它生成的是 code review 任务 prompt，不是审查报告本身。生成最终 prompt 时，删除不适用段落，不保留空占位符。

````text
你要在当前仓库 `{{仓库路径}}` 完成一次 code review。默认只输出审查报告和建议，不直接修改代码；除非用户明确要求修复，否则不要 apply patch。

【审查目标】

1. 项目类型：`{{项目类型}}`。
2. 审查主题：{{审查目标}}。
3. 重点判断：是否存在 bug、行为回归、契约漂移、错误处理风险、测试缺口、文档误导、发布风险或用户可见问题。
4. 不做事项：{{不直接修改代码的边界}}。

【审查范围】

请审查以下范围：

- `{{审查范围}}`

范围可能来自当前分支 diff、指定 commit、PR diff、用户指定文件或目录。不要把范围外的风格偏好包装成问题；如果发现范围外存在阻塞风险，可以单独列为“范围外风险”，并说明为什么需要关注。

【必读文件】

审查前必须先阅读：

- `{{项目规则文件}}`
- `{{README路径}}`
- `{{测试说明或开发日志}}`
- `{{审查范围相关文件}}`

【对比基准】

本次审查以以下基准为准：

- 基准类型：`{{当前分支 / 指定 commit / PR diff / 用户指定文件 / 发布包}}`
- 基准说明：{{对比基准说明}}
- 如果无法获取基准，必须说明无法获取的原因，并改用已读文件和当前 diff 做替代审查。

【风险地图生成要求】

不要直接套用固定审查清单。开始审查前，必须先基于项目规则、审查目标、对比基准和真实改动建立本轮风险地图，再从风险地图生成审查重点。

风险地图至少覆盖以下问题：

1. 本次改动改变了哪些承诺：{{项目核心承诺}}。
2. 哪些对象可能受影响：{{受影响对象}}。
3. 哪些项目不变量、状态关系、调用边界或文档口径必须保持不变：{{关键不变量}}。
4. 哪些失败模式最难发现、最难恢复，或者影响面最大：{{关键失败模式}}。
5. 哪些证据可以证明关键风险已经被覆盖：{{证据要求}}。
6. 用户或项目规则明确点名的高风险领域是什么：{{高风险领域}}。

可以参考项目常见风险维度来启发审查，例如用户承诺、接口与数据边界、状态流转、错误可观测性、证据充分性、发布或运行影响、长期维护成本、项目专项约束等。但这些只是启发，不是完整清单；最终 review prompt 必须要求执行者根据真实仓库补充风险，不得把示例当成审查范围上限。

如果某个示例维度与当前项目无关，生成 prompt 时必须删除；如果当前项目存在示例之外的关键风险，必须补充进去。

【证据链接规则】

1. 每条确认问题都必须给出可跳转 Markdown 相对路径位置；链接优先落到具体文件名，能定位到行号时写到行号。
2. 不要只写文件夹名代替关键证据，不要使用当前 IDE 或 Markdown 预览中无法跳转的绝对路径。
3. 如果只能定位到文件级别，必须说明无法给出行号的原因。
4. 证据必须来自代码、配置、文档、测试、日志、trace 或实际命令输出。
5. 没有证据的问题不要写成 finding，可以放入“开放问题”或“建议核对”。

【报告保存要求】

1. code review 报告必须保存为单独 Markdown 文档。
2. 如果用户指定了路径，保存到用户指定路径。
3. 如果用户没有指定路径，先根据项目习惯选择合理的 `{{报告目录}}`，例如项目已有 `docs/`、`reports/` 或类似目录，并在执行前说明路径。
4. 报告文件名应能体现任务主题和日期，但不能写死当前仓库命名。
5. 报告内所有文件链接必须使用可跳转 Markdown 相对路径交叉引用。

【输出格式】

最终输出必须 findings first，按严重程度排序：

```markdown
**审查发现**

**本轮风险地图**

- {{从真实项目和改动中提炼出的风险维度，不要原样复制模板示例}}

1. `{{严重级别}}`：{{问题标题}}
   - 位置：{{Markdown 相对路径:行号}}
   - 证据：{{代码、测试、日志或文档证据}}
   - 影响：{{用户影响、数据影响、发布影响或维护风险}}
   - 可能的解决方案：{{可行修复方向、涉及文件、预期影响和需要补充的验证；如果有多种方案，列 1 到 3 个方向，不要在证据不足时强行拍板}}
   - 验证建议：{{应该补跑或新增的测试}}
   - 是否阻塞：{{是否阻塞合并 / 发布 / 交付}}

**开放问题**

- {{无法从当前材料确认的问题}}

**测试缺口**

- {{未覆盖场景或未运行验证}}

**简短总结**

- {{一句话总结}}
```

严重级别参考：

- 阻塞：会导致功能不可用、数据丢失、发布失败、权限 / 隐私 / 安全高风险。
- 高：会造成主要用户路径回归、错误结果、难以恢复的状态污染或核心契约破坏。
- 中：边界场景错误、缺失关键验证、文档明显误导或可恢复的行为偏差。
- 低：局部可维护性风险、轻微文案问题或测试补强建议。

如果没有发现明确问题，也必须生成报告，写明“未发现阻塞项或明确缺陷”、已检查范围、剩余测试缺口和残余风险。不要为了凑 finding 把风格偏好写成 bug。

【注意】

1. 默认不修改代码。
2. 不要把审查 prompt 写成实现 prompt。
3. 不要只总结改了什么；必须优先找风险和证据。
4. 不要把历史测试结果写成本轮验证结果。
5. 如果需要后续修复，给出最小修复建议和验证建议，等待用户确认。
````

## 使用提醒

- 审查目标、范围和对比基准必须清楚；不清楚时要求执行者先只读核对。
- 发布审核、权限、隐私、远程请求等风险只在项目相关时启用。
- 如果用户要求“审查并修复”，prompt 也应先要求输出 findings，再进入修复闭环。
