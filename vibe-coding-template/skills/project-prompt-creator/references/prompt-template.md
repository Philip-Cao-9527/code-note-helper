# 项目任务 prompt 模板

本文件是 `project-prompt-creator` 的专属参考模板。生成最终 prompt 时，删除不相关段落，不保留空占位符；如果某些信息需要执行者现场核对，用“必须先核对”表达，不要编造事实。

````text
你要在当前仓库 `{{仓库路径}}` 完成一次“{{任务标题}}”任务。严格执行以下要求，不得跳过。

【硬性前置要求（必须先做）】

1. 不做任何修改前，必须先阅读并在进度中明确已读：
   - `{{项目规则文件}}`
   - `{{README路径}}`
   - `{{开发日志或架构文档}}`
   - `{{本轮相关调用链文件}}`
   - `{{用户指定必读文件}}`
2. 全程使用简体中文；代码、命令、报错原文、文件路径、字段名、域名、专有名词保留原文。
3. 先理解真实入口、调用链、数据结构、依赖顺序和验证方式，再做修改。
4. 不要回滚用户已有改动；遇到不相关 dirty 文件，记录并避开。
5. 不要把未验证写成已验证，不要把推测写成事实。
6. 修改文本文件时按项目要求处理编码；涉及中文文件时优先确认 UTF-8 可读。

【项目定位与边界】

1. 项目类型：`{{项目类型}}`。
2. 项目主目标：{{项目主目标}}。
3. 本轮任务范围：{{本轮任务范围}}。
4. 本轮明确不做：{{本轮不做什么}}。
5. 参考材料只用于学习组织方式和约束风格，不要机械复制私有路径、模块名、版本号、测试命令或发布规则：{{参考材料}}。

【版本与文档策略（必须遵守）】

1. 当前版本：`{{当前版本}}`。
2. 版本策略：{{版本策略}}。
3. 版本文件：`{{版本文件}}`。
4. README 版本位置：`{{README版本位置}}`。
5. 报告目录：`{{报告目录}}`。
6. 如果本轮不触发版本号或报告，最终总结必须明确说明原因。
7. 如果用户明确要求生成报告，必须按 `{{报告路径或命名规则}}` 生成，并使用可跳转 Markdown 相对路径交叉引用。链接优先落到具体文件名，能定位到行号时统一使用 `[文件名](相对路径#L行号)`；不要把 `:行号` 写进链接目标，不要只写文件夹名代替关键证据，不要使用当前 IDE 或 Markdown 预览中无法跳转的绝对路径。

【本次要完成的 TODO，必须全部落地】

A. {{事项 A 标题}}

1. 背景：{{背景 A}}
2. 定位要求：{{定位 A}}
3. 修改要求：{{修改 A}}
4. 验收标准：{{验收 A}}

B. {{事项 B 标题，可删除}}

1. 背景：{{背景 B}}
2. 定位要求：{{定位 B}}
3. 修改要求：{{修改 B}}
4. 验收标准：{{验收 B}}

C. {{事项 C 标题，可删除}}

1. 背景：{{背景 C}}
2. 修改要求：{{修改 C}}
3. 验收标准：{{验收 C}}

【实现约束】

1. 默认最小必要改动，但不是盲目保守；如果局部补丁无法闭环，先说明根因、必要性、影响范围、回归方案和可回滚边界，再做必要重构。
2. 代码生成必须基于真实仓库结构、真实调用链和真实测试入口。
3. 不要做无关重构，不要修改明确禁止修改的目录或文件：`{{禁止修改范围}}`。
4. 不要为了显得完整堆空壳、堆模板、堆不可验证约束。
5. 不新增没有依据的保护逻辑；保护逻辑包括但不限于固定超时、长度截断、条数上限、重试上限、静默降级、隐藏兜底、中断条件、容量边界、输入输出限制、重试 / 轮询策略、降级 / 回退策略、异常捕获策略和默认值策略。
6. 如确实需要新增保护逻辑，必须说明依据、触发时可见行为、误伤风险、验证方式和后续调整方式。
7. 错误处理必须显式暴露问题、便于排查；禁止 broad try/catch 后吞异常，禁止失败后伪造成功或返回空结果冒充正常。
8. 用户可见文案必须自然中文，不能暴露内部变量、函数名、模块名、堆栈或调试信息。
9. {{项目专项约束}}

【测试与验证要求，必须执行】

1. 运行 `{{验证命令 1}}`。
2. 运行 `{{验证命令 2}}`。
3. 对新增或修改的 Markdown / YAML / JSON / 文本文件做 UTF-8 可读性检查：`{{编码检查命令}}`。
4. 对 YAML、JSON、frontmatter 或配置文件做最小解析校验：`{{解析校验命令}}`。
5. 对文件结构和报告路径做真实存在性检查：`{{存在性检查命令}}`。
6. 如果本轮没有涉及某类语言、构建或运行入口，不要假装运行了对应语法、构建或单元测试；只列实际适用的校验。
7. 如果某项验证无法运行，必须说明：
   - 不可用工具或失败命令。
   - 为什么不可用。
   - 用了什么替代验证。
   - 仍未验证的边界。

【交付物要求】

最终输出按以下顺序组织：

1. 文件改动清单：逐文件说明改了什么。
2. 问题与修复闭环：说明定位、修改、复测和结果。
3. 模板化 / 设计质量提升说明：如果适用，说明哪些内容从硬编码改成占位符或条件段。
4. 版本同步清单：说明是否保持当前版本，是否修改版本相关位置。
5. 验证方式：列出实际运行命令和结果。
6. 证据路径：列出报告、日志、截图、trace、关键改动文件或命令输出证据；报告内文件链接必须优先使用可跳转 Markdown 相对路径；带行号时统一使用 `[文件名](相对路径#L行号)`。
7. 修复报告路径：`{{报告路径}}`，不触发时说明原因。
8. 风险与备注：区分已验证事实、工程判断和未验证边界。
9. 最终结论：是否达标，是否仍有边界风险。

【注意】

1. 不要把本任务扩展成无关业务开发。
2. 不要把参考项目的私有约束写进当前项目。
3. 不要把历史测试结果写成本轮验证结果。
4. 不要在没有证据时新增保护逻辑。
5. 不要保留空标题、英文占位说明、机器味模板句或不可复用硬编码。
````

## 使用提醒

- 如果用户要先计划再执行，改用 `$plan-mode-planner`。
- 如果用户要审查代码，改用 `$code-reviewer`。
- 如果用户要创建新 skill，改用 `$project-skill-creator`。
- 如果最终 prompt 需要交叉引用其他 skill，必须采用 `$skill-name` 形式，不要只写反引号包裹的 skill 名称。
- 如果任务触发报告或版本策略，优先引用项目 `AGENTS.md` 的真实规则，不要使用本模板里的示例值。
- 最终 prompt 必须整体放入一个 Markdown 文本块。
- 如果 prompt 内部包含三反引号，外层使用四反引号或更长围栏。
- 输出代码块前最多写一句中文引导；输出代码块后不要追加额外正文。
- 生成或改写 prompt 后，可运行 `python -X utf8 vibe-coding-template/skills/project-prompt-creator/scripts/validate_project_prompt.py path/to/generated-prompt.md` 检查硬性结构和关键质量规则；失败时按缺失项继续修改并复跑。
