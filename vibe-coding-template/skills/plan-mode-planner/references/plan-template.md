# Plan mode prompt 模板

本文件是 `plan-mode-planner` 的专属参考模板。它生成的是要求 Codex 使用 Plan mode 的 prompt，不是最终实现计划。生成最终 prompt 时，删除不适用段落，不保留空占位符。

````text
你要在当前仓库 `{{仓库路径}}` 处理一次“{{任务主题}}”任务。必须先使用 Plan mode，不允许跳过计划阶段直接实现。

【当前模式】

1. 本轮先进入 Plan mode。
2. 只允许做只读探索、阅读、搜索、静态分析和必要的非破坏性核对。
3. 不要编辑文件，不要 apply patch，不要新增报告，不要删除文件，不要修改版本号，不要运行会改变仓库状态的命令。
4. 不要一开始就写死详细技术方案；先读真实仓库和参考材料，再提炼需要用户拍板的关键决策。

【项目背景】

1. 项目类型：{{项目类型}}。
2. 项目目标：{{项目背景}}。
3. 本轮任务目标：{{任务目标}}。
4. 本轮明确不做：{{硬性边界}}。

【必读文件与参考材料】

开始提问或出计划前，必须先阅读并在进度里明确已读：

- `{{项目规则文件}}`
- `{{README路径}}`
- `{{开发日志或架构文档}}`
- `{{本轮相关文件}}`
- `{{参考材料}}`

参考材料只用于理解组织方式、约束风格或可迁移经验，不要机械复制私有路径、模块名、版本号、测试命令或发布规则。

【Plan mode 必须提炼的关键决策类型】

你需要从真实仓库和任务背景中提炼 2 到 4 个真正需要用户拍板的关键决策点。优先关注：

1. {{关键决策类型 1，例如架构边界、目录归属、是否拆分模块、是否改变公开契约}}
2. {{关键决策类型 2，例如版本/报告策略、兼容性取向、验证深度}}
3. {{关键决策类型 3，可删除}}
4. {{关键决策类型 4，可删除}}

不要把以下内容当成用户决策点：

- 能通过阅读仓库直接确认的事实。
- 单个函数名、变量名、文件名这类琐碎实现细节。
- 已经被用户硬性要求写死的边界。
- 只有一个合理答案的工程常识。

【每个决策点的输出要求】

每个决策点必须包含：

1. 为什么必须现在决策。
2. 不同取向分别影响什么。
3. 当前从仓库事实推导出的倾向。
4. 仍需要用户确认的选择。

你可以给出倾向，但不能替用户拍板。

【禁止一开始写死的技术细节】

在用户选择前，不要提前写死以下内容：

- {{禁止写死的技术细节 1}}
- {{禁止写死的技术细节 2}}
- {{禁止写死的技术细节 3，可删除}}

【用户选择后的收敛要求】

用户完成关键决策后，再输出可执行计划。计划必须覆盖：

1. 文件级改动范围。
2. 调用链或模板链路影响。
3. 测试与验证要求：{{实现后验证要求}}。
4. 版本、文档、报告策略；后续如生成报告或证据索引，文件链接必须优先使用可跳转 Markdown 相对路径。
5. 风险、回滚边界和未验证项。
6. 交付物顺序：{{交付物顺序}}。

【最终输出要求】

本轮 Plan mode 输出只做两件事：

1. 简要说明已读材料和当前事实。
2. 提出 2 到 4 个关键决策点，等待用户选择。

不要直接实现，不要输出最终补丁，不要生成修复报告，不要把计划写成已经完成的工作。
````

## 使用提醒

- 如果用户已经明确所有关键方向，可以让 Plan mode prompt 要求执行者快速确认事实后直接给计划，但仍不能跳过计划阶段。
- 如果任务只是普通执行 prompt，不需要用户拍板，改用 `project-prompt-creator`。
- 如果任务是 code review，改用 `code-reviewer`。
