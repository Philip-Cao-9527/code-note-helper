# 项目本地 skill 模板

本文件是 `project-skill-creator` 的专属参考模板。创建具体 skill 时，删除不相关段落，不保留空占位符。

## 推荐目录

按需创建，禁止空壳堆积。

```text
{{skill名称}}/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── references/
│   └── {{参考模板文件}}.md
├── scripts/
│   └── {{可复用脚本}}
├── templates/
│   └── {{可复制文件模板}}
└── assets/
    └── {{静态资源}}
```

最小结构只需要：

```text
{{skill名称}}/
└── SKILL.md
```

## 何时使用附加目录

- `references/`：有较长模板、固定输出骨架、审查清单、示例 prompt 或领域说明时使用。
- `scripts/`：有可重复运行的校验、生成、迁移或格式化脚本时使用。
- `templates/`：需要复制成目标文件的模板时使用。
- `assets/`：需要图片、示例数据、静态资源或二进制参考材料时使用。
- `agents/`：项目或平台需要 UI 元数据时使用。

不要为了看起来完整而创建空目录。

## SKILL.md 模板

````markdown
---
name: {{skill名称}}
description: {{一句话说明这个 skill 做什么、什么时候触发、适合哪些任务；不要写空泛口号。}}
---

# {{skill标题}}

## 技能定位

- 主职责：{{唯一主职责}}。
- 适用场景：{{适用场景}}。
- 不适用场景：{{不适用场景}}。
- 最终产物：{{最终产物}}。

## 与项目规则的关系

- 必须遵守：`{{项目规则文件}}`。
- 本 skill 只补充 {{流程名称}} 的专属流程，不覆盖项目级语言、编码、测试、版本、报告或安全规则。
- 如果本 skill 与项目规则冲突，以项目规则和用户最新指令为准。

## 必读文件

触发本 skill 后，先读取：

1. `{{项目规则文件}}`
2. `{{README路径}}`
3. `{{流程相关文件1}}`
4. `{{流程相关文件2}}`
5. `{{references文件，可删除}}`

不要无目标扫描整个仓库；只读取与本轮任务直接相关的材料。

## 输入信息

使用本 skill 前，尽量确认：

- 任务主题：{{任务主题}}
- 目标路径或范围：{{目标路径或范围}}
- 用户硬性边界：{{用户硬性边界}}
- 必做事项：{{必做事项}}
- 验证要求：{{验证要求}}
- 交付格式：{{交付格式}}

信息不完整时，优先从项目文件核对；无法确认时明确写“待确认”，不要编造。

## 使用流程

1. 确认用户目标和本 skill 是否适用。
2. 读取项目规则和必读文件。
3. 按 {{流程名称}} 的真实输入输出整理任务。
4. 必要时读取 `references/{{参考模板文件}}.md`。
5. 必要时运行 `scripts/{{脚本名}}`；运行前说明目的，运行后记录结果。
6. 生成最终产物或补丁方案。

## Skill 交叉引用

如果本 skill 需要引用其他 skill，必须使用 `$skill-name` 形式，例如 `$project-prompt-creator`。不要只写反引号包裹的 skill 名称，也不要写成普通文件名。

## 讲解类 skill 专项校验

当目标 skill 属于技术原理、八股讲解、从零教学、面试口述或项目追问类时，必须先读取 `$knowledge-explainer` 源模板，并保留其中的最高约束、1000 / 3000 字硬门槛、未达字数视为输出失败、输出结构、质量门槛、自检项和未实现能力边界。

生成目标 `SKILL.md` 后运行：

```powershell
python -X utf8 vibe-coding-template/skills/knowledge-explainer/scripts/validate_knowledge_explainer.py path/to/knowledge-explainer/SKILL.md
```

脚本返回非 0 时，继续修改目标 `SKILL.md` 并复跑，直到通过。这个脚本只校验结构和关键质量规则是否保留，不判断最终讲解内容是否真的写得好。

## 输出格式

最终输出必须包含：

1. {{输出项 1}}
2. {{输出项 2}}
3. {{输出项 3}}
4. {{验证结果}}
5. {{风险与未验证项}}

如果最终产物是 prompt，必须整体放入一个 Markdown 文本块；如果内部包含三反引号，外层使用四反引号或更长围栏。

如果最终产物包含报告、审查结果或证据索引，文件链接必须优先使用可跳转 Markdown 相对路径，链接优先落到具体文件名，能定位到行号时统一使用 `[文件名](相对路径#L行号)`，不要把 `:行号` 写进链接目标；不能只写文件夹名代替关键证据。

## 质量标准

- 产物必须具体、可检查、可落地。
- 不保留空标题、空占位符、英文占位说明或一次性任务残留。
- 不把其他项目的私有路径、模块名、版本号、测试命令或发布规则写成当前项目事实。
- 交叉引用其他 skill 时必须使用 `$skill-name` 形式。
- 不新增没有依据的保护逻辑；如确需加入中断条件、容量边界、输入输出限制、重试 / 轮询策略、降级 / 回退策略、异常捕获策略或默认值策略，必须说明依据、影响、可观测性、验证方式和后续调整方式。
- 不把未验证写成已验证。

## 验证方式

按任务类型运行：

- 结构检查：`{{结构检查命令}}`
- YAML / frontmatter 解析：`{{YAML或frontmatter校验命令}}`
- UTF-8 可读性：`{{编码检查命令}}`
- 项目专项验证：`{{项目专项验证命令}}`

如果验证不可用，必须说明不可用原因、替代验证和剩余风险。

## 自检

输出前检查：

1. 是否遵守 `{{项目规则文件}}`。
2. 是否没有编造项目事实。
3. 是否没有做超出本 skill 职责的事。
4. 是否明确验证方式和未验证边界。
5. 是否引用的 `references/`、`scripts/`、`templates/`、`assets/` 文件都真实存在。
````

## 验证清单

创建或更新 skill 后，至少执行：

1. 检查 `SKILL.md` frontmatter 是否存在 `name` 和 `description`。
2. 检查 skill 名称是否为小写字母、数字和连字符。
3. 检查 `agents/openai.yaml` 是否可解析；如果不存在且项目不需要，说明原因。
4. 检查 `references/`、`scripts/`、`templates/`、`assets/` 中被引用文件是否存在。
5. 用 UTF-8 读取所有新增或修改的 Markdown / YAML 文件。
6. 通读改动文件，确认没有空标题、英文占位说明、项目私有硬编码或一次性任务残留。
7. 如果创建或改写讲解类 skill，运行 `python -X utf8 vibe-coding-template/skills/knowledge-explainer/scripts/validate_knowledge_explainer.py path/to/knowledge-explainer/SKILL.md`，失败时继续修改并复跑。
