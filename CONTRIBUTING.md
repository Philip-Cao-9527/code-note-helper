# 贡献指南

感谢你关注 CodeNote Helper，也欢迎提交 Issue 和 Pull Request。

CodeNote Helper 是一个面向算法刷题、深度学习复盘、AI 辅助学习和长期记忆管理的浏览器扩展。由于浏览器扩展涉及页面注入、站点权限、用户数据、本地存储、远程 API、浏览器拓展商店审核等问题，任何改动都需要尽量保持清晰、克制、可维护。

项目当前重点方向请查看 README.md 的「后续开发计划」章节。较大的新功能建议先通过 Issue 讨论，再拆成小 PR 逐步实现。

---

## 提交 Issue 前

Bug 反馈请尽量提供可复现信息：

+ 扩展版本
+ 使用的浏览器，例如 Chrome / Edge
+ 浏览器版本
+ 操作系统
+ 出问题的页面链接
+ 具体操作步骤
+ 预期结果
+ 实际结果
+ 控制台报错截图或文字
+ 是否可以稳定复现

功能建议请尽量说明：

+ 这个功能解决什么问题
+ 具体使用场景是什么
+ 用户会在哪个页面触发这个功能
+ 是否需要新增权限或远程访问
+ 是否会影响现有功能
+ 是否有类似工具或截图可以参考

对于新站点适配、主题系统、知识库、云盘同步、手机端提醒等较大功能，建议先开 Issue 讨论方案，不建议直接提交大范围 PR。

---

## 提交 Pull Request 前

Pull Request 请尽量保持小而清晰。

一个好的 PR 应该满足：

+ 只解决一个明确问题
+ 改动范围尽量小
+ 不混入无关格式化、重构或样式调整
+ 不无故修改 `manifest.json`、`service-worker.js`、权限、注入顺序等敏感文件
+ 有清楚的改动说明
+ 有复现步骤或测试说明
+ 用户可见文案使用自然中文
+ 不引入乱码、BOM 混乱、换行风格混乱等问题

不太适合直接合并的 PR：

+ 一个 PR 同时改主题、样式、service worker、manifest、题单导入和日志系统
+ 大范围重构但没有说明必要性
+ 修改了很多无关文件
+ 新增权限但没有解释为什么必须新增
+ 新增远程访问但没有说明数据流向
+ 没有复现步骤，也没有测试方式
+ 直接提交大段生成代码，但没有人工检查和验证

---

## 浏览器扩展相关注意事项

修改下面这些内容时，请在 PR 中说明原因：

+ `manifest.json`
+ `service-worker.js`
+ `content_scripts`
+ `host_permissions`
+ `permissions`
+ `shared/*`
+ `content/*`
+ `popup/*`
+ `options/*`
+ `notes/*`

请不要无故扩大权限范围，尤其不要轻易改成：

+ `<all_urls>`
+ `*://*/*`
+ `https://*/*`

确实需要新增站点或权限时，请说明：

+ 为什么必须新增
+ 是否可以使用现有权限完成
+ 是否可以收窄到更精确的域名或路径
+ 是否可以改为用户主动触发后再执行
+ README、隐私政策、商店说明是否需要同步更新

---

## 站点适配要求

当前项目强调站点隔离。不同站点的选择器、抓取逻辑、初始化逻辑不要混在一起。

新增或修改站点适配时，请注意：

+ LeetCode CN / LeetCode US / CodeFun2000 / Deep-ML / TorchCode / ChatGPT / Gemini / Claude 等逻辑应保持隔离
+ 不要让某个站点的 DOM 选择器影响其他站点
+ 不要为了兼容一个页面而破坏已有页面
+ 修改 content script 注入顺序时必须说明原因
+ 新增站点时需要同步 README 中的支持站点说明
+ 新增站点时需要说明是否涉及新的权限或审核风险

---

## 远程 API、云盘和消息提醒

项目允许用户主动配置自己的 LLM API、WebDAV、云盘或消息提醒服务，但远程返回内容只能作为数据处理，不能作为代码执行。

禁止：

+ 引入远程托管 JavaScript
+ 动态插入远程 `<script>`
+ 使用 `eval`
+ 使用 `new Function`
+ 执行远程返回的字符串
+ 未经用户明确配置就把数据发送到第三方服务
+ 把用户数据发送到开发者服务器但不说明

新增 Google Drive、OneDrive、飞书 Bot、手机端提醒等能力时，请在 PR 中说明：

+ 数据会发送到哪里
+ 什么时候发送
+ 是否由用户主动触发
+ 是否经过开发者服务器
+ 用户如何关闭
+ 用户如何撤销授权
+ README 和隐私政策是否需要同步更新

## 错误处理要求

错误处理应该帮助定位真实问题，而不是把问题藏起来。

不推荐：

+ 空的 `catch`
+ 捕获异常后直接返回空结果
+ 失败后伪装成成功
+ 用户界面直接显示内部变量名、函数名、原始堆栈
+ 大量刷屏式 `console.log`

推荐：

+ 控制台保留必要技术日志
+ 用户界面显示自然中文提示
+ 区分用户取消、网络失败、状态码错误、服务端异常、页面结构变化
+ Bug 修复优先解决根因，而不是靠静默兜底掩盖问题

---

## 不要无依据新增限制

请不要随意加入限制。

谨慎新增：

+ 固定超时
+ 文本长度截断
+ 条数上限
+ 重试上限
+ 轮询次数上限
+ 静默降级
+ 隐藏兜底

确实必须新增限制时，请在 PR 中说明：

+ 为什么必须有这层限制
+ 触发条件是什么
+ 对用户有什么影响
+ 是否会误伤正常内容
+ 是否可以配置或后续调整
+ 是否已经测试

尤其是题目描述、题解、参考答案、Prompt、AI 流式输出等内容，不要无依据截断。展示层可以折叠，数据层不要随意裁掉。

---

## 用户可见文案要求

所有用户可见文字应使用自然、友好的中文。

包括：

+ 按钮文案
+ Toast 提示
+ 错误信息
+ 空状态提示
+ 设置项说明
+ 弹窗文案
+ 悬浮提示

避免：

+ 英文调试语句
+ 内部变量名
+ 函数名
+ 原始报错堆栈
+ 人机味很重的表达
+ 普通用户看不懂的开发术语

技术细节可以放在控制台日志、Issue 讨论或 PR 说明里，不要直接暴露给普通用户。

---

## 测试建议

涉及页面行为的改动，请尽量提供测试说明，**优先采用控制台脚本注入法测试页面运行效果**。

建议覆盖：

+ 首次加载
+ 二次加载
+ 刷新页面
+ 页面切换后重新进入
+ DOM 延迟出现
+ 重复初始化
+ 空数据
+ 长文本
+ 设置开启 / 关闭
+ 成功返回
+ 失败返回
+ 网络异常
+ 滚动边界
+ 长列表或长对话
+ 不同站点之间切换

使用了 Playwright、Playwright MCP 或其他浏览器自动化工具时，欢迎在 PR 中附上测试步骤。

## AI 生成代码说明

可以使用 AI 或 Agent 工具辅助开发，但提交前请务必人工检查。

请确认：

+ 你理解提交的代码
+ 你确认没有混入无关改动
+ 你本地运行和测试过
+ 没有引入远程代码、动态执行或权限扩大
+ 没有加入无依据的限制
+ 没有破坏现有功能和架构边界

维护者会按最终代码质量和测试情况审查 PR。

---

## 维护者可能关闭 PR 的情况

以下情况可能会被关闭：

+ 长期没有回应
+ 改动范围过大且不愿拆分
+ 与项目当前方向不一致
+ 没有复现信息
+ 无法验证
+ 引入权限或审核风险但没有说明
+ 混入大量无关改动
+ 已经由其他 commit 或 PR 修复
+ 代码无法维护

关闭 PR 不代表否定贡献本身。很多想法可以先通过 Issue 继续讨论，或者拆成更小的 PR 后重新提交。
