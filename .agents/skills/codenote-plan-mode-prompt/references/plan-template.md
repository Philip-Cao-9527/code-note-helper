# CodeNote Helper Plan Mode Prompt 模板

本文件是 `codenote-plan-mode-prompt` 的专属参考模板。生成最终 prompt 时，用真实任务信息替换尖括号字段；删除不适用段落，不保留空标题。

````text
你要在当前仓库 `<仓库路径>` 处理一次“<任务主题>”任务。必须先使用 Plan mode，不允许跳过计划阶段直接实现。

【当前模式】

1. 本轮先进入 Plan mode。
2. 只允许做只读探索、阅读、搜索、静态分析和必要的非破坏性核对。
3. 不要编辑文件，不要 apply patch，不要新增报告，不要删除文件，不要修改版本号，不要运行会改变仓库状态的命令。
4. 不要一开始写死完整技术方案；先读真实仓库和参考材料，再提炼需要用户拍板的关键决策。

【项目背景与边界】

1. CodeNote Helper 是 Chrome MV3 浏览器扩展，主线功能包括算法刷题笔记、间隔复习、深度学习复盘和 AI 对话时间轴。
2. 本轮任务目标：<任务目标>。
3. 本轮明确不做：<硬性边界>。
4. 默认简体中文、Windows 与 UTF-8 编码。
5. 默认保持当前版本；是否升版必须来自用户明确要求和现场版本核对。

【必读文件与参考材料】

开始提问或出计划前，必须先阅读并在进度里明确已读：

- `AGENTS.md`
- 如果 `AGENTS.override.md` 存在，也一并读取；如果不存在，明确说明不存在。
- `DEVLOG.md`
- `README.md`
- `<本轮相关文件或报告>`
- `<参考材料>`

参考材料只用于理解组织方式、约束风格或可迁移经验，不要机械复制私有路径、模块名、版本号、测试命令或发布规则。

【Plan mode 必须提炼的关键决策类型】

你需要从真实仓库和任务背景中提炼 2 到 4 个真正需要用户拍板的关键决策点。优先关注：

1. 架构边界、目录归属、职责拆分或是否改变公开契约。
2. 版本与报告策略、是否同版本补丁、验证深度和证据要求。
3. 权限、远程请求、用户数据、Chrome Web Store 审核成本或隐私说明一致性。
4. 是否需要从局部补丁升级为更大范围治理，以及回滚颗粒度。

不要把以下内容当成用户决策点：

- 能通过阅读仓库直接确认的事实。
- 单个函数名、变量名、文件名这类琐碎实现细节。
- 已经被用户硬性要求写死的边界。
- 只有一个合理答案的工程常识。

【每个决策点的输出要求】

每个决策点必须包含：

1. 为什么必须现在决策。
2. 不同取向分别影响什么。
3. 当前从仓库事实推导出的倾向。
4. 仍需要用户确认的选择。

你可以给出倾向，但不能替用户拍板。

【用户选择后的收敛要求】

用户完成关键决策后，再输出可执行计划。计划必须覆盖：

1. 文件级改动范围。
2. 入口、调用链、模板链路或注入顺序影响。
3. 测试与验证要求。
4. 版本、`DEVLOG.md`、docs 修复报告和证据链接策略。
5. 权限审核、用户数据、远程请求和用户可见文案风险。
6. 回滚边界和未验证项。

【最终输出要求】

本轮 Plan mode 输出只做两件事：

1. 简要说明已读材料和当前事实。
2. 提出 2 到 4 个关键决策点，等待用户选择。

不要直接实现，不要输出最终补丁，不要生成修复报告，不要把计划写成已经完成的工作。

如果最终 prompt 需要交叉引用其他 skill，必须采用 `$skill-name` 形式，例如 `$codenote-fix-prompt`；不要只写反引号包裹的 skill 名称。
````
