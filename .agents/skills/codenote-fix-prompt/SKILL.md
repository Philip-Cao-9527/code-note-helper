# CodeNote Helper Prompt 生成技能

生成可直接复制使用的任务 prompt，不直接开始改代码。

## 先做什么

1. 先确认这次是“生成 prompt”，不是“直接执行修复”。
2. 先提取输入参数：
   - 仓库路径
   - 目标版本
   - 是否同版本补丁
   - 任务背景 / 问题现象
   - 必做 TODO
   - 必测范围
   - 交付格式
3. 若用户没有补全全部参数，按仓库默认约束补足，但要避免编造事实。

## 默认补全规则

- 仓库路径：默认使用当前仓库绝对路径。
- 版本策略：若用户未明确要求升版，默认写成“在当前版本内继续做最小必要增量修复”，不要自动升级到下一个版本。
- 必读文件：默认至少包含：
  - `AGENTS.override.md`
  - `DEVLOG.md`
  - `README.md`
- 历史修复文档：
  - 若任务明显涉及回归风险，补上最近相关的 `docs/fix-report-v*.md`
  - 只引用和本次任务直接相关的报告，不要堆太多
- 代码范围：默认要求阅读与任务相关的真实调用链，例如 `content/`、`popup/`、`shared/`、`notes/`、`options/`
- 语言：全程简体中文，保留代码、命令、报错原文、路径、API 字段名原文

## Prompt 结构

按下面顺序组织生成结果，除非用户明确要求更短版本，否则不要删掉这些骨架：

1. 任务开场
2. `【硬性前置要求（必须先做）】`
3. `【版本与文档策略（必须遵守）】`
4. `【本次要完成的 TODO，必须全部落地】`
5. `【实现约束】`
6. `【测试要求，必须执行】`
7. `【交付物要求】`
8. `【注意】`

使用 `references/prompt-template.md` 作为底稿，按场景删减不相关段落，不要保留空标题或空占位符。

## 最终输出包裹规则

- 最终生成的 prompt 必须整体放在 Markdown 文本块中，方便用户一键复制。
- 输出代码块前最多写一句中文引导；输出代码块后不要追加额外正文，避免影响复制。
- 默认外层使用 ` ```text ` 包裹；若 prompt 内部包含三反引号代码块，外层必须改用 ` ````text ` 包裹，并用 ` ```` ` 结束，确保整个 prompt 不会被内部代码块提前截断。
- 不要用 `<prompt>`、`</prompt>` 或其他 XML/HTML 风格标签替代 Markdown 文本块。

## 任务类型加料规则

### 页面抓取 / DOM / 注入 / 浏览器行为

- 强制写明：先复现，再定位，再修复。
- 强制要求使用 Playwright MCP 或等价浏览器自动化验证并留证据。
- 强制要求给出：
  - 页面 URL
  - 页面真实可见内容
  - 插件抓取结果
  - console / trace / 截图 / harness 路径
- 禁止在没有证据前直接拍脑袋改 selector、写死 DOM 路径或引入远程兜底。

### popup / notes / options / shared 数据链路

- 强调真实调用链、状态结构、依赖顺序和注入顺序。
- 明确要求不要新造平行状态源，优先复用现有 state、store、record 结构。
- 必写防回归点，例如：
  - `popup` 分页或筛选不能牵连其他页面
  - `notes`、`options`、`shared` 数据结构不能漂移
  - `review-fsrs.js` 等依赖顺序不能破坏

### 题单导入 / 静态映射

- 默认优先要求“内置静态映射”。
- 明确禁止运行时访问 GitHub README、远程解析 Markdown 或新增 `github.com` 权限。
- 要求说明链接核对策略、去重策略和剩余风险。

### 日志 / 调试 / 自动入库 / 复习链路

- 允许新增控制台日志，但只用于开发排查。
- 明确要求：
  - 日志前缀统一
  - 输出结构化可读字段
  - 不输出裸 `Object`
  - 不暴露到用户可见 UI
- 禁止为了日志加入固定超时、固定截断或静默降级。

### 权限 / manifest / 远程请求 / 审核友好

- 若任务触及 `manifest.json`、`permissions`、`host_permissions`、`content_scripts.matches`、远程请求域名、README 隐私说明，必须加入审核友好自检。
- 必写：
  - 为什么必须新增权限 / 域名
  - 是否能收窄范围
  - 是否能改为用户显式触发
  - 是否会影响 Chrome Web Store 审核
- 明确禁止：
  - `<all_urls>`
  - `*://*/*`
  - 远程托管代码
  - `eval` / `new Function`

### 审核模式 / 发布前审计 / 代码风险报告

- 若用户要求“审核”“审计”“review”“发布前检查”“Chrome Web Store 风险”“代码风险报告”，默认生成审核模式 prompt。
- 审核模式默认只要求输出报告，不直接修复代码；除非用户明确要求修复，否则 prompt 中要写明“发现阻塞发布问题时只列证据、影响、建议修复方案，不擅自改业务代码”。
- 审核报告默认输出到 `docs/`，并要求按风险级别分组：阻塞发布 / 高风险 / 中风险 / 低风险 / 未发现风险。
- 必须要求审核报告中的所有问题位置、风险证据、代码引用使用可跳转 Markdown 文件链接，方便从报告直接跳到代码：
  - 推荐格式：`[相对路径:行号](<仓库绝对路径/相对路径:行号>)`
  - Windows 路径包含空格时，链接目标必须用尖括号包住，例如 `[shared/api-client.js:42](<D:/path with spaces/code-note-helper/shared/api-client.js:42>)`
  - 每条确认风险至少给一个可跳转到具体行号的代码链接；如果只能定位到文件级别，必须说明无法给出行号的原因，并给文件级链接。
  - 不要只写纯文本路径；不要只写文件名不写行号；不要把不可跳转路径当成证据。
- 审核模式 prompt 的交付物中必须要求报告包含“风险清单表”，字段至少包括：级别、可跳转位置、证据摘要、影响、建议处理方式、是否阻塞发布。
- 若审核涉及 manifest、权限、远程请求、隐私政策或商店说明，必须要求每个风险项同时检查“代码实际行为”和“README / 隐私政策 / CHANGELOG / manifest 声明”是否一致。

## 风格要求

- 语气要像“执行指令”，不是泛泛建议。
- 约束要写得具体，可检查，可落地。
- TODO 多时按 `A / B / C / D` 分块；单任务就保留一个任务块。
- “问题现象 / 定位要求 / 修复要求 / 实现约束 / 测试要求 / 交付物要求”这些层级尽量稳定，方便复用。
- 如果用户已经给了强约束，优先保留原话中的硬性边界，再做模板化归纳。

## 输出前自检

输出 prompt 前，逐条检查：

1. 是否明确这是“生成 prompt”，不是已经开始改代码。
2. 是否明确目标版本以及“同版本补丁 / 升版”策略。
3. 是否明确要求先读 `AGENTS.override.md`、`DEVLOG.md`、`README.md`。
4. 是否明确禁止修改 `references/`。
5. 是否明确“先复现再修复”的证据要求（如果任务涉及浏览器行为）。
6. 是否明确 Node 测试与浏览器验证都要执行（如果任务涉及代码变更）。
7. 是否避免加入没有依据的固定超时、截断、重试上限和静默兜底。
8. 是否要求最终输出包含文件改动、证据路径、闭环结论。
9. 如果是审核模式，是否要求审核报告中的问题位置使用可跳转 Markdown 文件链接，并且尽量精确到行号。
10. 是否已把最终 prompt 整体包裹在 Markdown 文本块中；若内部包含三反引号，外层是否已改用四反引号文本块；是否没有使用 `<prompt>` 标签；代码块后是否没有追加额外正文。

## 推荐输入格式

当用户只给一个模糊想法时，先把信息整理成下面这些字段，再生成 prompt：

```text
任务主题：
目标版本：
是否升版：
问题背景：
必做 TODO：
相关模块：
强制约束：
必测场景：
交付格式：
```

如果用户已经提供了完整长 prompt，就不要重写业务内容，只做“抽象模板化”和“提炼成可复用结构”。
