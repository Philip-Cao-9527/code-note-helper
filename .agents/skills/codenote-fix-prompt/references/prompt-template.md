# CodeNote Helper 普通任务 Prompt 模板

本文件是 `codenote-fix-prompt` 的专属参考模板。生成最终 prompt 时，用真实任务信息替换尖括号字段；无法确认的事实写成“必须先核对”，不要编造。删除不适用段落，不保留空标题。

````text
你要在当前仓库 `<仓库路径>` 完成一次“<任务主题>”任务。严格按普通执行模式推进：先只读理解真实结构，再做最小必要修改、验证和交付；不要停在方案讨论阶段。

【硬性前置要求（必须先做）】

1. 修改前必须先阅读并在进度里明确已读：
   - `AGENTS.md`
   - 如果 `AGENTS.override.md` 存在，也一并读取；如果不存在，明确说明不存在，不要写成必读事实。
   - `DEVLOG.md`
   - `README.md`
   - `<本轮相关调用链、测试文件、报告或文档>`
2. 全程使用简体中文；代码、命令、报错、路径、字段名、域名、专有名词保留原文。
3. 先理解真实入口、调用链、数据结构、依赖顺序和验证方式，再做修改。
4. 不要回滚用户已有改动；遇到无关 dirty 文件，记录并避开。
5. 不要把未验证写成已验证，不要把推测写成事实。
6. 注意 Windows 与 UTF-8 编码，修改文本文件时保持原文件风格。

【项目定位与边界】

1. CodeNote Helper 是 Chrome MV3 浏览器扩展，主线功能包括算法刷题笔记、间隔复习、深度学习复盘和 AI 对话时间轴。
2. 本轮范围：<本轮任务范围>。
3. 本轮明确不做：<禁止修改范围或不做事项>。
4. 根目录 `references/` 中标注“仅供参考，非项目代码”的内容不要直接修改。

【版本与文档策略（必须遵守）】

1. 修改前现场核对 `manifest.json`、`popup/popup.html`、`README.md` 中的当前版本。
2. 版本策略：<保持当前版本或按用户要求升版>。
3. 需要更新 `DEVLOG.md` 时，只记录本轮真实改动、验证和边界。
4. 需要新增 docs 修复报告时，使用 `docs/fix-report-vX.Y.Z-YYYYMMDD-主题.md`，不要覆盖已有报告。
5. 报告内文件链接使用可跳转 Markdown 相对路径，优先落到具体文件名，能定位到行号时写行号。

【本次要完成的事项，必须全部落地】

A. <事项 A 标题>

1. 背景：<问题背景或目标>
2. 定位要求：<必须先核对的真实入口、调用链或数据结构>
3. 修改要求：<最小必要改动要求>
4. 验收标准：<可检查结果>

B. <事项 B 标题，可删除>

1. 背景：<问题背景或目标>
2. 修改要求：<最小必要改动要求>
3. 验收标准：<可检查结果>

【实现约束】

1. 优先最小必要改动，不做无关重构。
2. 不破坏站点隔离、脚本注入顺序、共享数据结构和既有调用链。
3. 不无故新增权限、扩大 `host_permissions` 或 `content_scripts.matches`。
4. 不引入远程托管代码、`eval`、`new Function` 或不可审计动态执行。
5. 不新增无依据的固定超时、文本截断、条数上限、重试上限、静默回退或吞异常。
6. 用户可见文案必须自然中文，不暴露内部变量、函数名、模块名、堆栈或调试信息。
7. <本轮专项约束>

【测试与验证要求，必须执行】

1. 运行与本轮改动直接相关的 Node 测试或静态检查：`<测试命令>`。
2. 如果涉及页面行为、交互、滚动、接口返回或浏览器扩展运行时链路，使用 Playwright MCP 或等价浏览器工具验证并留证据。
3. 如果只修改文档、prompt 或本地 skill，不擅自补 Playwright；改用结构、编码、frontmatter、链接和 git 范围检查。
4. 如果某项验证无法运行，必须说明失败命令、失败原因、替代验证和剩余风险。

【交付物要求】

最终输出按以下顺序组织：

1. 文件改动清单：逐文件说明改了什么。
2. 问题与修复闭环：说明定位、修改、复测和结果。
3. 版本同步清单：说明是否保持当前版本，是否改动版本相关位置。
4. 验证方式：列出实际运行命令和结果。
5. 证据路径：列出关键文件、报告、日志、截图、trace 或命令证据。
6. docs 修复报告路径：不触发时说明原因。
7. 风险与未验证边界：区分已验证事实、工程判断和未验证项。
8. 最终结论：是否达标。

【注意】

1. 不要把本任务扩展成无关业务开发。
2. 不要把历史测试结果写成本轮验证结果。
3. 不要在没有证据时新增保护逻辑。
4. 如果遇到登录、人机验证、验证码等必须人工完成的步骤，暂停并说明边界。
````

## 使用提醒

- 需要先决策再实现时，改用 `$codenote-plan-mode-prompt`。
- 需要生成审查任务 prompt 时，改用 `$codenote-code-review-prompt`。
- 需要创建或更新本地 skill 时，改用 `$codenote-skill-creator`。
- 如果最终 prompt 需要交叉引用其他 skill，必须采用 `$skill-name` 形式，不要只写反引号包裹的 skill 名称。
- 最终 prompt 必须整体放入一个 Markdown 文本块；如果内部包含三反引号，外层使用四反引号或更长围栏，避免复制时被截断。
- 生成或改写 prompt 后，可运行 `python -X utf8 .agents/skills/codenote-fix-prompt/scripts/validate_codenote_fix_prompt.py path/to/generated-prompt.md` 检查硬性结构和关键质量规则；失败时按缺失项继续修改并复跑。
