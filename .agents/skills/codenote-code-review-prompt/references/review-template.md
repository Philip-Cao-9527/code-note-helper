# CodeNote Helper Code Review Prompt 模板

本文件是 `codenote-code-review-prompt` 的专属参考模板。生成最终 prompt 时，用真实审查范围替换尖括号字段；删除不适用段落，不保留空标题。

````text
你要在当前仓库 `<仓库路径>` 完成一次 CodeNote Helper code review。默认只输出审查报告和建议，不直接修改代码；除非用户明确要求“审查并修复”，否则不要 apply patch。

【审查目标】

1. 项目类型：Chrome MV3 浏览器扩展。
2. 审查主题：<审查主题>。
3. 重点判断：是否存在 bug、行为回归、契约漂移、错误处理风险、测试缺口、文档误导、权限审核风险、用户数据风险或用户可见问题。
4. 不做事项：<不直接修改代码的边界>。

【审查范围】

请审查以下范围：

- `<审查范围>`

范围可能来自当前分支 diff、指定 commit、PR diff、发布包、用户指定文件或目录。不要把范围外的风格偏好包装成问题；如果发现范围外存在严重风险，可以单独列为“范围外风险”，并说明为什么需要关注。

【必读文件】

审查前必须先阅读：

- `AGENTS.md`
- 如果 `AGENTS.override.md` 存在，也一并读取；如果不存在，明确说明不存在。
- `DEVLOG.md`
- `README.md`
- `<审查范围相关文件、测试、配置或历史报告>`

【外部事实核对要求】

如果审查范围涉及外部平台行为、浏览器 API、OAuth / Google Drive / WebDAV、权限 / 审核政策、第三方 API、最新文档、跨浏览器差异、已知社区问题或反复定位失败的 bug，必须先调用 `$web-search` 建立外部证据链。

code review 不能只看本地 diff。对外部 API / 平台行为相关问题，必须优先使用官方文档、浏览器厂商文档、供应商文档、官方 GitHub / release notes 或维护者讨论；高质量社区讨论只能作为补充，不能作为唯一依据。审查报告中需要区分已确认事实、工程推断和仍需真实浏览器 / 真实账号 / 真实 OAuth / 真实审核环境验证的边界。

【风险地图生成要求】

不要直接套用固定审查清单。开始审查前，必须先基于项目规则、审查目标、对比基准和真实改动建立本轮风险地图，再从风险地图生成审查重点。

风险地图至少覆盖：

1. 本次改动改变了哪些用户承诺、数据承诺、权限承诺或维护承诺。
2. 哪些站点、页面、共享模块、存储结构、注入顺序或文档口径可能受影响。
3. 哪些失败模式最难发现、最难恢复，或者影响面最大。
4. 哪些证据可以证明关键风险已经被覆盖。
5. 哪些风险来自 Chrome MV3、权限与 host 范围、远程请求、用户数据、站点隔离、README / 隐私政策 / manifest 一致性、用户可见中文文案或测试证据不足。
6. 哪些风险依赖外部平台行为或最新文档，是否已经通过 `$web-search` 建立可追溯来源。

上述风险域只是启发，不是完整清单；必须根据真实 diff 和项目规则补充本轮专属风险。

【证据链接规则】

1. 每条确认问题都必须给出可跳转 Markdown 相对路径位置；链接优先落到具体文件名，能定位到行号时使用 `#L{行号}` 锚点，例如 `[content.js](../src/content.js#L120)`；不要把 `:120` 写进链接目标。
2. 不要只写文件夹名代替关键证据，不要使用当前 IDE 或 Markdown 预览中无法跳转的绝对路径。
3. 如果只能定位到文件级别，必须说明无法给出行号的原因。
4. 证据必须来自代码、配置、文档、测试、日志、浏览器验证记录或实际命令输出。
5. 没有证据的问题不要写成 finding，可以放入“开放问题”或“建议核对”。

【报告质量要求】

1. 审查报告必须保持 findings first 和严重程度排序；新手友好解释只是每条 finding 内的辅助理解，不能放在 findings 前面，也不能弱化 bug、风险、证据和验证建议。
2. 每条 finding 的“影响”“为什么这是问题”“可能的解决方案”后面，应尽量补一小段通俗解释，说明这件事对 CodeNote Helper 的扩展功能、用户数据、权限审核或用户体验会造成什么真实后果。复杂 finding 可以用 2 到 4 个短段或短列表拆开，但不要写成长篇教程。
3. 通俗解释借鉴本仓库 `.agents/skills/knowledge-explainer/SKILL.md` 第二部分 `详细原理讲解（通俗版）` 的方法：从零讲清楚、把术语翻译成项目里发生了什么、用短小例子帮助理解。这里只借鉴风格和质量要求，不套用 3000 字硬性字数，也不要把 review 报告改成知识讲解。
4. 技术字段、manifest 字段、API 字段名、函数名、文件名和浏览器 API 名称可以保留英文原文，但不要把多个字段名连续堆成难懂大段。必要时先用中文解释业务含义，再列原字段名。
5. 浏览器扩展术语必须配中文含义和用户影响。例如 `host_permissions` 是“扩展被允许访问哪些网站”，`content_scripts.matches` 是“脚本会自动注入到哪些页面”；OAuth、Google Drive、WebDAV 风险要解释成“用户数据会不会离开本地”“凭据会不会泄露”“同步失败用户能不能看懂原因”等可理解后果。
6. 避免整段英文、长串专业名词或只由字段名串起来的解释。不能只写“manifest 字段不一致导致 permission drift”，要补充“简单说，就是扩展实际能访问的页面和文档承诺不一致，用户或审核员会不知道它到底会读哪些网站”这类说明。

【报告保存要求】

1. code review 报告必须保存为单独 Markdown 文档。
2. 如果用户指定路径，保存到用户指定路径。
3. 如果用户没有指定路径，优先保存到 `docs/`，文件名包含审查主题和日期。
4. 报告内所有文件链接必须使用可跳转 Markdown 相对路径交叉引用；带行号时统一使用 `[文件名](相对路径#L行号)` 范式。

【输出格式】

最终输出必须 findings first，按严重程度排序：

```markdown
**审查发现**

**本轮风险地图**

- 从真实项目和改动中提炼出的风险维度。

### `严重级别`：问题标题

位置：Markdown 相对路径链接；能定位时用 `[文件名](相对路径#L行号)` 写到行号。

证据：代码、测试、日志或文档证据。需要多条证据时用短段落或紧凑列表组织，不要堆成难读的大段。

影响：用户影响、数据影响、发布影响或维护风险。说明为什么这是问题，而不只是代码现象。

可能的解决方案：可行修复方向、涉及文件、预期影响和需要补充的验证；如果有多种方案，用 1 到 3 个短小标题或编号方向区分，不要在证据不足时强行拍板。

验证建议：应该补跑或新增的测试。

是否影响合并 / 发布 / 交付：说明是否会影响合并、发布或交付。

**开放问题**

- 无法从当前材料确认的问题。

**测试缺口**

- 未覆盖场景或未运行验证。

**简短总结**

- 一句话总结。
```

严重级别参考：

- 严重：会导致功能不可用、数据丢失、发布失败、权限 / 隐私 / 安全高风险。
- 高：会造成主要用户路径回归、错误结果、难以恢复的状态污染或核心契约破坏。
- 中：边界场景错误、缺失关键验证、文档明显误导或可恢复的行为偏差。
- 低：局部可维护性风险、轻微文案问题或测试改进建议。

如果没有发现明确问题，也必须生成报告，写明“未发现严重项或明确缺陷”、已检查范围、剩余测试缺口和残余风险。不要为了凑 finding 把风格偏好写成 bug。

【注意】

1. 默认不修改代码。
2. 不要把审查 prompt 写成实现 prompt。
3. 不要只总结改了什么；必须优先找风险和证据。
4. 不要把历史测试结果写成本轮验证结果。
5. 如果需要后续修复，给出最小修复建议和验证建议，等待用户确认。
6. 如果最终 prompt 需要交叉引用其他 skill，必须采用 `$skill-name` 形式，例如 `$codenote-fix-prompt`；不要只写反引号包裹的 skill 名称。
````
